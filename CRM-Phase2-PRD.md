## 1. 需求概述

### 1.1 背景与目标
一期已完成「用户中心 / 订单中心」的数据可查（看得清）。二期解决「销售 leads 够得着」与「权限管得住」两个核心诉求：

- **销售中心**：承接注册线索的自动沉淀、分配与人工跟进转化，并**接入外呼系统**（点号码拨打 + 通话录音 + AI 通话小结 + 自动归档），形成「注册 → 分配 → 跟进/外呼 → 付费回流用户中心」的转化闭环；销售组长可按权重自动分配、超时掉库、重新分配。
- **系统配置·角色权限**：建立 RBAC（角色-权限）体系，权限细分到子权限（无 / 只读 / 可操作），配合业务线数据隔离与操作日志，支撑多角色、多业务线安全协作。

### 1.2 目标用户（内部 B 端角色）
| 角色 | 二期核心诉求 | 高频模块 |
| --- | --- | --- |
| 💼 销售专员 | 领取「未付费-未体验+有手机号」线索、记录跟进、外呼促成付费 | 销售中心（待领取 / 我的跟进 / 通话记录） |
| 👔 销售组长（Leader） | 配置自动分配比例 / 掉库规则、重新分配线索、查看组内全部跟进 | 销售中心（含设置） |
| 🎧 客服 / 运营 | 辅助查看与跟进 | 销售中心（只读/操作，视权限） |
| ⚙️ 管理员 / 系统配置 | 配置角色权限矩阵、成员账号与数据范围、审计操作日志 | 系统配置·角色权限 |

### 1.3 上线目标与依赖
- **目标上线**：8/7。
- **关键依赖**：外呼接入方案（第三方外呼平台对接方式、录音回传、AI 小结生成能力）。**前端开发工时与外呼相关测试量需在外呼方案明确后评估；若方案确定较晚，8/7 可能顺延。**
- **工时估算（参考）**：服务端 8 pd；前端待定（依赖外呼方案）；测试约 3 天（不含外呼外场景，外呼工作量待定）。

## 2. 模块一：销售中心

### 2.1 业务流转（漏斗口径）
销售中心**仅处理具备触达条件（有手机号）且尚未付费未体验**的潜在客户线索。

```
用户注册 → 有手机号 自动进入销售中心 → 待领取 → 领取跟进（跟进中） → 已付费 → 回流用户中心
```

| 场景 | 归属 | 说明 |
| --- | --- | --- |
| **注册（未付费-未体验）+ 有手机号** | **销售中心线索** | 自动进入「待领取」线索池 |
| 注册 + 无手机号（谷歌邮箱 / Facebook / AppID 第三方登录） | 用户中心 | 拿不到手机号，无法电话跟进，仅在用户中心留档 |
| 已体验 / 已付费 / 付费逾期 | 用户中心 | 全部在用户中心管理 |
| 跟进中转为「已付费」 | **回流用户中心** | 用户状态改写为「付费」，离开销售中心 |

> **手机号规则**：仅 `kakao` / `手机号` 登录方式带手机号；`谷歌邮箱` / `Facebook` / `AppID` 无手机号。**不支持手动录入手机号**，线索来源全部由系统自动判定沉淀。

### 2.2 跟进进度定义
| 进度 | 含义 | 标签色 |
| --- | --- | --- |
| **待领取** | 线索已进入销售中心，尚无销售认领 | 橙 |
| **跟进中** | 已被销售领取，正在联系与转化 | 蓝 |
| **暂不跟进** | 暂时搁置（如家长仍在观望）；仍归属该销售 | 灰 |
| **已付费** | 转化成功，用户状态改写为「付费」，回流用户中心 | 绿 |

> 「更新跟进」弹窗可选进度为：跟进中 / 已付费 / 暂不跟进。

### 2.3 页面结构：三个 Tab

顶部为流程说明横幅（可展开「查看详情」）+ 全局筛选区（业务线筛选 `LineFilter` + 关键字搜索 +「通话记录」Tab 额外提供通话结果筛选）。页面右上角在有 `sales_config` 权限时显示「设置」入口。

#### Tab 1 · 待领取注册用户（pool）
- 展示尚无领取人的线索（`isPoolLead`：是销售线索且 `salesOwner` 为空）。
- Tab 标题带数量：`待领取注册用户 (N)`。
- 支持关键字搜索（手机号 / 用户ID / 姓名 / 国家 / 渠道来源 / 最新备注）+ 业务线筛选。
- 操作列：**领取跟进**（需 `sales_claim` 权限）。点击后线索 `salesOwner` 置为当前用户、进度置「跟进中」，写入一条跟进记录（备注：「销售已领取，开始跟进已注册用户」），自动跳转到「我的跟进」。

#### Tab 2 · 我的跟进（follow）
- 展示本人已领取线索（`isClaimedLead`）；**超管 / 拥有重新分配权限的组长可见范围内全部领取记录**（`seeAllOwners`）。
- 组长身份（拥有 `sales_config`）在此 Tab 顶部展示提示横幅：「你是销售组长，可查看并重新分配组内（业务线范围）全部销售的跟进线索。」
- 支持关键字搜索 + 业务线筛选。
- 相较待领取列表，额外列：**最新备注 / 最后更新时间 / 领取人**。
- 操作列（按权限组合展示）：
  - **外呼**（`sales_dial`）：无手机号时按钮禁用。
  - **更新跟进**（`sales_update`）。
  - **重新分配**（`sales_reassign`）：仅组长/超管。

#### Tab 3 · 通话记录（calls）
- 集中查看历史外呼记录；Tab 标题带数量。
- 顶部横幅：「外呼系统与 CRM 轻量集成：坐席点击号码发起外呼，挂断后填写通话小结，系统自动归档到客户档案与销售跟进记录。」
- 支持关键字（手机号 / 用户ID / 客户 / 备注）+ 通话结果筛选（已接通 / 无人接听）+ 业务线筛选。
- **数据隔离**：非超管（无重新分配权限）仅见本人坐席（`agent`）的记录。

页面底部展示统计提示：当前销售中心共 N 条线索。

### 2.4 关键交互流程

**① 领取线索**
在「待领取注册用户」Tab 按关键字/业务线筛选，点「领取跟进」认领 → 自动进入「我的跟进」，进度置「跟进中」。

**② 常规跟进（更新跟进弹窗）**
在「我的跟进」点「更新跟进」→ 弹窗展示手机号、领取人（只读）+ 本次备注（必填）；底部为**跟进记录时间轴**（进度 · 备注 · 时间 · 领取人，含外呼录音播放条）。保存后写入一条新的跟进记录快照。
> 说明：原型「更新跟进」弹窗以备注为主并写入历史；进度枚举（跟进中/已付费/暂不跟进）为业务约定，落地实现时以进度选择驱动转化闭环。

**③ 外呼跟进（外呼弹窗，两阶段）**
1. **呼叫阶段**：弹窗显示客户姓名 + 业务线 + 手机号，「通话中」状态并实时计时（`mm:ss`）；按钮为「挂断并填写小结」。
2. **小结阶段**：挂断后进入小结表单——
   - 通话结果：已接通 / 无人接听（无人接听时长记为「—」）。
   - 通话时长：自动带入（只读）。
   - **通话小结（必填）**：接通且时长>0 时，系统模拟 **AI 自动总结**（约 1.5s 生成，展示「AI正在为您总结通话记录...」，生成文案可编辑）。
   - 保存后：生成一条通话记录（`CallRecord`），并**自动归档到该线索的销售跟进记录**（备注前缀「【外呼自动记录】」）；接通的通话附带一条**录音链接**，可在跟进记录时间轴内播放。

**④ 重新分配（组长/超管）**
在「我的跟进」点「重新分配」→ 弹窗显示当前领取人 + 选择新销售（下拉为可分配销售，排除当前领取人）→ 确认后改写 `salesOwner`，并归档一条跟进记录（备注：「重新分配给 XXX」）。

**⑤ 转化闭环**
本次跟进选「已付费」→ 用户状态改写为「付费」，该用户离开销售中心，在用户中心可查。

### 2.5 销售组长设置（自动分配 / 掉库）
入口：页面右上角「设置」（需 `sales_config` 权限）。

- **超时掉库规则**：开关（默认开启）+「领取后超过 `N` 小时无跟进记录，自动掉库并重新按比例分配」（`N` 默认 24，范围 1–720 小时）。
- **线索分配比例**：以表格列出所有可被分配的销售（销售 + Email + 权重），权重范围 0–100（**填 0 表示不分配**）。当线索自动分配或掉库重分配时，按权重随机分配给相关销售。
- 「可被分配的销售」= 账号状态为「启用」且其角色的销售模块（`sales`）权限为「可操作」。

### 2.6 列表字段（数据字典）

**待领取 / 我的跟进列表（复用用户中心-二期字段）**
| 字段 | 说明 |
| --- | --- |
| 用户ID | 复用用户中心字段（固定左列） |
| 姓名 | 本地名优先，无则英文名 |
| 用户状态 | 未付费-未体验 / 未付费-体验中 / 未付费-已体验 / 付费 / 付费逾期（按课时记录动态判定） |
| 用户类型 | 正式用户 / 测试用户（按手机号规则自动识别，剔除测试线索） |
| 年龄段 | 3-5 / 6-8 / 9-12 / 13-17 / 18+ |
| 登录账号 | 邮箱 / FB / kakao ID / AppID / 手机号 |
| 业务线 | 由渠道归因推导 |
| 渠道来源-落地页 | 落地页投放归因文案 |
| 渠道 code | 落地页渠道 code |
| 渠道来源-App | 三方归因（投放渠道 / 子渠道） |

**通话记录字段（`CallRecord`）**
| 字段 | 说明 |
| --- | --- |
| 起呼时间 | 发起外呼的时间 |
| 客户 / 手机号 / 业务线 | 被呼叫线索的客户姓名、号码、所属业务线 |
| 通话结果 | 已接通 / 无人接听 |
| 时长 | `mm:ss`；无人接听为「—」 |
| 本次跟进记录 | 坐席填写的通话小结（必填），同步写入销售跟进记录 |

## 3. 模块二：系统配置·角色权限

菜单标题「系统配置 · 角色权限」，含三个 Tab：**角色权限 / 成员账号 / 操作日志**。

### 3.1 权限模型（RBAC）

**权限级别（`PermLevel`）**
| 级别 | 含义 | 效果 |
| --- | --- | --- |
| `none` 无权限 | 菜单隐藏 | 不可见 |
| `view` 只读 | 查看 / 导出 | 只读 |
| `operate` 可操作 | 查看 + 新增 / 编辑 | 可操作 |

**数据权限（`DataScope`）**
| 范围 | 含义 |
| --- | --- |
| `all` 全部业务线 | 可见全部业务线数据 |
| `line` 指定业务线 | 仅可见分配到的业务线数据 |

**权限矩阵：主模块 + 子权限（细分到子权限）**
| 主模块 | 子权限 |
| --- | --- |
| 用户中心（users） | 修改用户（users_edit） |
| 销售中心（sales） | 领取（sales_claim）、外呼（sales_dial）、更新跟进（sales_update）、重新分配（sales_reassign）、销售设置（sales_config） |
| 订单中心（orders） | —（`orders` 不提供「可操作」，仅 无/只读） |
| 系统配置（system） | 新增角色（system_role_add）、编辑角色（system_role_edit）、删除角色（system_role_delete）、新增成员（system_acc_add）、编辑成员（system_acc_edit） |
| 用户中心-二期（usersV2）`三期` | — |
| 渠道管理（channels）`三期` | 新建/编辑/删除/生成code/渠道参数 |
| 落地页（landing）`三期` | 新建 / 删除 |
| 商品包（packages）`三期` | 新建 / 编辑 / 上下架 |
| 优惠券（coupons）`三期` | 新建 / 延长 / 停止 / 编辑 |

> 权限矩阵中标注「三期」的模块灰显占位，二期不启用但保留于矩阵中以便统一维护。

**权限联动规则**
- 主模块权限切换为**非「可操作」**时，其下所有子权限自动重置为 `none`。
- 编辑角色时，子权限行仅在其父模块为「可操作」时才展示可配置。
- 子权限只有 无 / 可操作 两档；主模块有 无 / 只读 / 可操作 三档（`orders` 无「可操作」）。

### 3.2 Tab 1 · 角色权限
- **角色列表**：角色名（内置角色带「内置」标签、后期角色带「预定」标签）、职责描述、数据范围（全部业务线 / 指定业务线）、成员数、操作（编辑权限 / 删除）。
  - 内置角色不可删除；被成员占用的角色不可删除（提示「有 N 个成员正在使用」）。
- **权限矩阵表**：行=模块（主模块 + 子模块平铺缩进），列=各角色，单元格显示该角色对该模块的权限级别标签。图例：可操作 = 查看+新增/编辑；只读 = 查看/导出；无权限 = 菜单隐藏。
- **新增 / 编辑角色弹窗**：角色名称（必填，≤20 字）、职责描述、数据范围（单选：全部业务线 / 指定业务线）、权限配置表（逐模块选 无/只读/可操作，子权限随父模块联动）。保存写入操作日志。

**内置角色（种子数据）**
| 角色 | 数据范围 | 说明 |
| --- | --- | --- |
| 超级管理员（默认兜底） | 全部业务线 | 未匹配到账号角色时按超管处理，全模块可操作 |
| 管理员 / 系统配置 | 全部业务线 | 账号鉴权、业务线/模版、角色权限 |
| 市场投放 / 增长 | 指定业务线 | 渠道管理、落地页 |
| 运营 / 商业化 | 指定业务线 | 商品包、优惠券、落地页 |
| 客服 / 用户支持 | 指定业务线 | 用户中心、订单中心 |
| 销售组长 | 指定业务线 | 管理销售团队线索分配与掉库规则 |
| 学习服务 / 班主任 | 指定业务线 | 后期随服务节点引入（预定） |

### 3.3 Tab 2 · 成员账号
- **列表字段**：姓名、登录邮箱、角色、数据范围（角色为 `all` 显示「全部业务线」；`line` 显示已分配业务线标签，未分配显示「—」）、状态（启用/停用开关，需编辑权限）、最近登录、操作（编辑）。
- **新增 / 编辑成员弹窗**：
  - 姓名（必填）。
  - 登录邮箱（必填，需合法邮箱；限 `@dinoai.ai`，占位 `name@dinoai.ai`）。
  - 角色（必填，单选）。
  - 数据范围（业务线，多选）：角色数据范围为 `all` 时禁用并显示「全部业务线」；为 `line` 时**必填**。
  - 状态（启用 / 停用开关，默认启用）。
- 启用/停用切换、新增/编辑成员均写入操作日志。

### 3.4 Tab 3 · 操作日志（审计）

- 分页展示（默认 15 条/页）。

## 4. 权限与数据隔离（全局规则）

### 4.1 菜单可见性（二期）
| 模块 | 销售专员 | 销售组长 | 客服/运营 | 管理员 | 超管 |
| --- | --- | --- | --- | --- | --- |
| 销售中心 | 可操作（本人范围） | 可操作 + 设置/重分配 | 可查看/操作（视配置） | 可操作 | 全部 |
| 系统配置·角色权限 | 不可见 | 不可见 | 不可见 | 可操作 | 可操作 |

### 4.2 数据隔离原则
- **业务线隔离**：数据范围为 `line` 的账号仅见其分配业务线的数据；`all`（超管/管理员）见全业务线。
- **销售中心归属隔离**：非超管销售仅见本人领取的线索与本人外呼记录；超管 / 拥有 `sales_reassign` 的组长可见业务线范围内全部线索与通话记录。
- **登录鉴权**：仅 `@dinoai.ai` 邮箱可登录/注册。
- **兜底策略**：登录邮箱未匹配到成员账号时，原型按「超级管理员」处理（保证可用）；正式实现应改为按最小权限或拒绝访问，落地时确认。

## 5. 数据结构摘要

- `Student.salesOwner / salesProgress / salesLatestNote / salesUpdatedAt / salesHistory[]`：销售中心跟进字段。
- `SalesFollowLog { progress, note, time, owner, audioUrl? }`：单条跟进记录（含录音）。
- `CallRecord { id, studentId, customer, phone, businessLine, result, duration, note, agent, time }`：外呼记录。
- `SalesSettings { autoDropEnabled, autoDropHours, allocations[{email,weight}] }`：组长分配/掉库设置。
- `Role { id, name, desc, builtin, planned?, dataScope, perms:Record<ModuleKey,PermLevel> }`：角色。
- `Account { id, email, name, roleId, businessLines[], status, lastLogin?, salesLead? }`：成员账号。
- `AuditLog { id, time, actor, module, action, target? }`：操作日志。
