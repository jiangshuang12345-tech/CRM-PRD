Dino English 运营平台（CRM）产品需求文档

本文档基于「CRM 与运营平台需求对齐」会议纪要，按业务诉求分期落地。一期聚焦基础查询能力（用户中心、订单中心），用户归因仅区分应用商店来源（App Store / Google Play）与国家；二期建设渠道体系并升级商业化配置能力，并将渠道、商品包、优惠券、落地页收拢到「营销中心」菜单下，另含用户中心二期与角色权限。文档内嵌可交互原型，可逐模块体验。

背景与目标

Dino English 面向海外多业务线（韩国 / 沙特 / 马来 / 越南 等）开展投放与运营。当前业务依赖人工散表查询用户与订单、渠道归因口径不统一、促销配置依赖研发。CRM 一期目标是先把「用户、订单查得到、看得清」做扎实，让运营与销售同学自助完成日常查询（此时仅按应用商店来源 + 国家做粗归因）；二期再建设渠道分级体系，把用户中心升级为按「业务线 · 渠道」精准归因，并把商品（商品包）与促销（优惠券、落地页）配置能力交还运营，减少对研发的依赖。

一期价值

  • 用户 / 订单一处可查，替代散表
  • 用户按应用商店来源 + 国家粗归因
  • 运营 / 销售按多条件秒级定位用户/订单

二期价值

  • 渠道结构标准化、可生成投放 code
  • 用户中心升级为「业务线 · 渠道」精准归因
  • 商品包自助上下架 + 满减券（KOL 多码结算）
  • 落地页一键打通渠道/商品/促销；角色权限 + 数据隔离 + 审计

衡量指标

  • 用户/订单查询自助率 ≥ 90%
  • 渠道归因覆盖率 ≥ 95%（二期）
  • 促销配置不再依赖研发

产品定位详述

平台不是一个孤立的「查询工具」，而是随业务成熟度逐层演进的运营中台：先把分散在各处的运营数据收拢为「单一事实来源」，再把商业化配置能力交还运营，最终以用户为中心做精细化触达与服务。其价值主线始终围绕「降低对研发的依赖、提升一线自助效率、让运营动作可归因可衡量」。

是什么

  • 海外多业务线的运营中台
  • 用户 / 渠道 / 商品主数据中心
  • 运营动作的统一操作台

解决什么

  • 散表查询、口径不统一
  • 归因断链、促销依赖研发
  • 触达与服务无抓手、难闭环

不做什么（边界）

  • 不替代 App / 学习产品本身
  • 不做面向 C 端用户的前台
  • 初期不做复杂 BI / 数仓（对接为主）

📍
当前使用方（暂定）：运营 + 销售。一期主要由运营与销售使用（用户中心 / 订单中心查询）；下列为平台完整角色画像（含市场、管理员、学习服务等后续 / 潜在角色，并对应系统配置的内置角色），随能力上线逐步引入。

🛠️ 运营 / 商业化运营 当前使用方

负责商品上下架、促销活动与用户运营动作；一期高频查询用户与订单。

核心诉求：快速查到用户/订单、看清状态与有效期；二期商品/促销自助配置、不依赖研发；后期做触达与服务编排。
高频模块：用户中心 · 订单中心 ·（二期）商品包 / 优惠券 / 落地页 ·（后期）触达/服务节点

💼 销售 / 业务跟进 当前使用方

负责意向 / 付费用户的跟进、转化、续费与对账核查。

核心诉求：按多条件快速查到用户与订单、看清付费状态与有效期，支撑转化跟单、续费提醒与对账；后期可单点触达跟进。
高频模块：用户中心 · 订单中心 ·（二期看）优惠券/落地页转化 ·（后期）单点触达

📣 市场投放 / 增长 后续

负责海外各业务线的拉新投放与渠道归因。

核心诉求：渠道结构清晰、能生成投放 code、投放链接一键可得、归因可追。
高频模块：渠道管理 · 落地页管理 ·（看）用户/订单归因

🎧 客服 / 用户支持 后续

响应用户咨询、处理订单与账号问题（后续纳入使用方）。

核心诉求：按多条件快速查到用户与订单、看清状态与有效期；后期可单点触达答疑与关怀。
高频模块：用户中心 · 订单中心 ·（后期）单点触达/服务节点

⚙️ 管理员 / 系统配置 后续

维护角色权限、成员账号、业务线/模版与数据范围。

核心诉求：账号鉴权（限 @dinoai.ai）、按职能配角色权限、数据按业务线隔离、操作可审计。
高频模块：系统配置 · 角色权限 · 成员账号 · 操作日志

🧑‍🏫 学习服务 / 班主任 后期

随「服务节点」上线而引入，负责体验期跟进、续费引导与流失挽回。

核心诉求：按生命周期节点拿到待办、对单个用户触达与跟进、记录服务结果。

后期能力前瞻：单点触达 & 服务节点

📨 单点触达用户

从用户中心选定单个用户或人群，触发 push / 站内信 / 短信 / EDM / 客服外呼等触达动作。

  • 场景：体验转化、到期续费提醒、流失召回、关怀通知
  • 抓手：复用已有「用户状态（注册/体验中/体验逾期/付费/付费逾期）」与业务线/渠道维度做精准圈选
  • 受益角色：运营、客服、学习服务

🧩 增加服务节点

在用户生命周期上编排服务节点，形成 SOP 与任务流转，把触达沉淀为可跟踪的服务动作。

  • 场景：体验期跟进、付费引导、续费提醒、流失挽回、学习陪伴
  • 抓手：以订单有效期 / 到期时间驱动节点触发，任务可分派、可记录结果
  • 受益角色：学习服务/班主任、客服、运营
🔄
后期能力与一/二期天然衔接：一期沉淀的用户状态与归因数据是触达的圈选依据，二期的商品包/优惠券是触达与服务节点的转化承接，形成「查得到 → 配得了 → 够得着」的完整运营闭环。

迭代计划 / 路线图

按「先查询、后配置」的节奏分两期交付，先解决业务每天都要用的查询场景，再交付运营自助配置能力。下列时间为相对排期建议（可按实际研发资源调整）。

一期基础查询能力建议 3~4 周

目标：运营 / 销售可自助查询用户与订单信息，替代线下散表。一期暂无渠道体系，用户归因仅区分应用商店来源（App Store / Google Play）与国家。

👤
用户中心（一期） P0
按 用户ID/姓名/登录账号 搜索；按 注册渠道（App Store/Google Play）、国家、状态筛选；支持 5 种注册方式（含非手机号注册）与登录账号、到期时间、最近修改人展示；可修改基本资料。
🧾
订单中心 P0
按 订单ID/用户ID/商品 搜索，国家、订单状态、支付方式筛选；展示有效期到期时间。
🌐
平台能力 P0
多语言（英 / 韩 / 阿 / 中，默认英文）切换；登录限 @dinoai.ai 邮箱。

一期交付物 / 验收

  • 用户、订单列表查询、筛选、分页可用；字段口径与本 PRD 数据字典一致。
  • 用户中心可按 应用商店来源 + 国家 + 状态 检索；可修改基本资料并记录最近修改人。
  • 登录鉴权（@dinoai.ai 邮箱验证码）可用。
二期渠道体系与商业化配置能力建议 4~5 周

目标：建设渠道分级体系并把用户中心升级为「业务线 · 渠道」精准归因；把商品（商品包）、促销（满减券 + 多码结算）与投放（落地页）的配置能力交还运营，形成「归因 → 配置 → 投放 → 发放 → 核销 → 结算」闭环。渠道、商品包、优惠券、落地页统一归入「营销中心」菜单，用户中心二期与角色权限独立陈列。

🛍️
营销中心（菜单分组） 新增
将渠道管理 / 商品包 / 优惠券 / 落地页 收拢为「营销中心」一级菜单下的子菜单，集中承载市场投放与商业化配置动作。
🌿
渠道管理 P1 营销中心
业务线 → 渠道类型 → 一/二/三级渠道树；任意层级生成可复制的渠道 code（7 位随机码）。
👥
用户中心 · 二期 P1
在一期基础上把「应用商店 + 国家」升级为「业务线 · 渠道 · 渠道 code」精准归因，支持业务线全局筛选。
📦
商品包管理 P1 营销中心
新增/编辑/详情/上下架；业务线、币种、价格、有效期。
🎟️
优惠券管理（多码结算） P1 营销中心
满减券配置 + 一券多优惠码（按 KOL），独立统计使用量用于结算；支持管理优惠码/批量复制/编辑/延长/停止发放。
🔗
落地页管理 P1 营销中心
一键关联渠道（code）+ 商品包 +（可选）优惠码，按业务线模版生成投放落地页链接，可预览/复制/删除。
🛡️
系统配置 · 角色权限 P1 新增
RBAC：角色权限矩阵（无/只读/可操作）、成员账号与业务线数据权限、操作日志；权限全平台生效。

二期交付物 / 验收

  • 渠道树可增删改、可生码并复制；用户注册渠道 code 与渠道树打通。
  • 用户中心 · 二期按「业务线 · 渠道」归因，支持业务线全局筛选。
  • 商品包可上下架，下架后不可被券绑定（前置校验）。
  • 满减券可生成多优惠码、按 KOL 统计使用量；可调整可用商品、延长领取时间、停止发放（二次确认）。
  • 券与商品包通过 商品包ID 关联，币种与业务线一致。
  • 落地页可基于已配置业务线模版生成可用投放链接，并关联到对应渠道/商品包/优惠码。
  • 角色权限按模块分级生效：菜单过滤、操作按钮显隐、数据按业务线隔离；角色与成员变更可在操作日志中追溯。

  • 可用商品必须为已存在的商品包，按商品包 ID 校验，不存在则报错。
  • 停止发放后不可继续领取；领取截止时间可向后延长以恢复「已生效」。
  • 券币种、业务线与所绑商品包须一致（前置约束）。

2.4 落地页管理 二期 P1 新增模块

一键生成投放落地页链接，关联渠道、商品包与优惠券（优惠码）；不同业务线使用各自的落地页模版生成对应链接，作为投放与归因的统一出口。

用户故事

作为市场投放同学，我希望选择业务线、渠道（已生成 code）、商品包、（可选）优惠券与具体 KOL 优惠码，再设投放有效期，一键生成可直接投放的落地页链接并复制，无需手工拼参数。

2.6 系统配置 · 角色权限 二期 P1 新增模块

面向内部 B 端使用者的角色与权限配置：定义每个角色可访问的功能模块及操作级别，并为成员账号分配角色与数据权限（业务线）范围；角色与成员账号的变更留痕可审计。

用户故事

作为管理员，我希望按职能配置角色（每个模块设 无权限 / 只读 / 可操作）、为成员分配角色与可见业务线范围，并能查看谁在什么时间做了什么操作，以便权限最小化、数据隔离、操作可追溯。

关键业务规则

  • 内置角色不可删除；角色下仍有成员时不可删除（需先转移成员）。
  • 成员数据权限随其角色的数据范围生效：「指定业务线」必须至少选择一个业务线。
  • 无某模块权限时该菜单不可见，直接访问其路由会重定向到第一个可访问页面。
  • 「只读」仅能查看/导出，「可操作」才显示新增/编辑等操作入口。
  • 操作日志当前记录角色与成员账号的变更（新增/编辑/删除角色、新增/编辑/启停成员）；用户、商品包、优惠券、落地页等业务模块的操作暂不计入。
