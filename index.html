背景与目标

Dino English 面向海外多业务线（韩国 / 沙特 / 马来 / 越南 等）开展投放与运营。当前业务依赖人工散表查询用户与订单、渠道归因口径不统一、促销配置依赖研发。CRM 一期目标是先把「用户、订单查得到、看得清」做扎实，让运营与销售同学自助完成日常查询（此时仅按应用商店来源 + 国家做粗归因）；二期再建设渠道分级体系，把用户中心升级为按「业务线 · 渠道」精准归因，并把商品（商品包）与促销（优惠券、落地页）配置能力交还运营，减少对研发的依赖。

一期价值

  • 用户 / 订单一处可查，替代散表
  • 用户按应用商店来源 + 国家粗归因
  • 运营 / 销售按多条件秒级定位用户/订单

二期价值

  • 渠道结构标准化、可生成投放 code
  • 用户中心升级为「业务线 · 渠道」精准归因
  • 商品包自助上下架 + 满减券（KOL 多码结算）
  • 落地页一键打通渠道/商品/促销；角色权限 + 数据隔离 + 审计

衡量指标

  • 用户/订单查询自助率 ≥ 90%
  • 渠道归因覆盖率 ≥ 95%（二期）
  • 促销配置不再依赖研发

产品定位详述

平台不是一个孤立的「查询工具」，而是随业务成熟度逐层演进的运营中台：先把分散在各处的运营数据收拢为「单一事实来源」，再把商业化配置能力交还运营，最终以用户为中心做精细化触达与服务。其价值主线始终围绕「降低对研发的依赖、提升一线自助效率、让运营动作可归因可衡量」。

是什么

  • 海外多业务线的运营中台
  • 用户 / 渠道 / 商品主数据中心
  • 运营动作的统一操作台

解决什么

  • 散表查询、口径不统一
  • 归因断链、促销依赖研发
  • 触达与服务无抓手、难闭环

不做什么（边界）

  • 不替代 App / 学习产品本身
  • 不做面向 C 端用户的前台
  • 初期不做复杂 BI / 数仓（对接为主）

三期能力：销售 leads（外呼）· 单点触达 & 服务节点

三期核心是销售中心（分配 / 跟进 leads + 外呼），原型已就绪，详见 3.1 销售中心；下列为同属三期的触达与服务能力前瞻。

📨 单点触达用户 三期

从用户中心选定单个用户或人群，触发 push / 站内信 / 短信 / EDM / 客服外呼等触达动作。

  • 场景：体验转化、到期续费提醒、流失召回、关怀通知
  • 抓手：复用已有「用户状态（注册/付费/付费逾期）」「销售中心线索跟进 / 外呼记录」与业务线/渠道维度做精准圈选
  • 受益角色：运营、客服、学习服务

🧩 增加服务节点 三期

在用户生命周期上编排服务节点，形成 SOP 与任务流转，把触达沉淀为可跟踪的服务动作。

  • 场景：体验期跟进、付费引导、续费提醒、流失挽回、学习陪伴
  • 抓手：以订单有效期 / 到期时间驱动节点触发，任务可分派、可记录结果
  • 受益角色：学习服务/班主任、客服、运营

四期能力前瞻：本地化配置（文案 + 物料）

🌏 本地化配置平台迁入 CRM 四期

现有一个独立平台可供业务自助修改本地化文案与物料；四期目标是把该平台整体迁入 CRM，与渠道 / 落地页 / 触达能力打通，统一账号、权限与业务线口径。

  • 能力：按业务线 / 语言维护界面与投放文案；维护各地区物料（图片 / Banner / 素材）供落地页与触达复用
  • 价值：业务自助改文案/物料，减少对研发与外部工具依赖；本地化与投放链路一体化
  • 受益角色：运营、市场投放、本地化 / 内容
🔄
各阶段天然衔接：一期沉淀的用户状态与归因数据是触达的圈选依据，二期的商品包/优惠券/销售跟进是触达与服务节点的转化承接，四期的本地化文案/物料又回哺渠道与落地页，形成「查得到 → 配得了 → 够得着 → 改得动」的完整运营闭环。
