Dino English 运营平台（CRM）产品需求文档

本文档基于「CRM 与运营平台需求对齐」会议纪要，按业务诉求分期落地。一期聚焦基础查询能力（渠道、用户基本信息、订单信息），二期聚焦商业化配置能力（商品包、优惠券）。文档内嵌可交互原型，可逐模块体验。

🔄
本次同步原型变更：① 课包统一更名为「商品包」（全平台导航、字段、文案）；② 优惠券支持一券多优惠码（按 KOL），分别统计使用量用于结算，新增「管理优惠码」与批量复制；③ 新增「落地页管理」模块（二期）：一键关联渠道 + 商品包 + 优惠码生成投放链接，按业务线模版拼装。
前序变更：渠道树顶层「业务线」+7 位随机 code；用户中心列与搜索调整（到期时间/最近修改人）；订单中心新增所属业务线与有效期；多语言（中/英/韩/阿）与业务线全局筛选。详见各模块「本期更新」。

背景与目标

Dino English 面向海外多业务线（韩国 / 沙特 / 马来 / 越南 等）开展投放与运营。当前业务依赖人工散表查询用户与订单、渠道归因口径不统一、促销配置依赖研发。CRM 一期目标是先把「查得到、看得清、能归因」做扎实，让市场与客服同学自助完成日常查询；二期再把商品（商品包）与促销（优惠券）配置能力交还运营，减少对研发的依赖。

一期价值

  • 渠道结构标准化、可生成投放 code
  • 用户注册按渠道 code 归因
  • 用户 / 订单一处可查，替代散表

二期价值

  • 商品包（商品）运营自助上下架
  • 满减券 + 一券多码（KOL 结算）
  • 落地页一键打通渠道/商品/促销

衡量指标

  • 用户/订单查询自助率 ≥ 90%
  • 渠道归因覆盖率 ≥ 95%
  • 促销配置不再依赖研发

迭代计划 / 路线图

按「先查询、后配置」的节奏分两期交付，先解决业务每天都要用的查询场景，再交付运营自助配置能力。下列时间为相对排期建议（可按实际研发资源调整）。

一期基础查询能力建议 3~4 周

目标：渠道结构标准化并可生码归因；客服 / 运营可自助查询用户基本信息与订单信息，替代线下散表。

🌿
渠道管理 P0
业务线 → 渠道类型 → 一/二/三级渠道树；任意层级生成可复制的渠道 code（7 位随机码）。
👤
用户基本信息查询 P0
按 用户ID/姓名/登录账号 搜索，业务线、状态筛选；支持 5 种注册方式（含非手机号注册）与登录账号、到期时间、最近修改人展示；可修改基本资料。
🧾
订单信息查询 P0
按 订单ID/用户ID/商品 搜索，业务线、订单状态、支付方式筛选；展示所属业务线与有效期到期时间。
🌐
平台能力 P0
多语言（中 / 英 / 韩 / 阿）切换；业务线筛选全局共享联动；登录限 @dinoai.ai 邮箱。

一期交付物 / 验收

  • 渠道树可增删改、可生码并复制；用户注册渠道 code 与渠道树打通。
  • 用户、订单列表查询、筛选、分页可用；字段口径与本 PRD 数据字典一致。
  • 登录鉴权（@dinoai.ai 邮箱验证码）可用。
二期商业化配置能力建议 3~4 周

目标：把商品（商品包）、促销（满减券 + 多码结算）与投放（落地页）的配置能力交还运营，与订单 / 商品打通，形成「配置 → 投放 → 发放 → 核销 → 结算」闭环。

📦
商品包管理 P1
新增/编辑/详情/上下架；业务线、币种、价格、有效期。
🎟️
优惠券管理（多码结算） P1
满减券配置 + 一券多优惠码（按 KOL），独立统计使用量用于结算；支持管理优惠码/批量复制/编辑/延长/停止发放。
🔗
落地页管理 P1 新增
一键关联渠道（code）+ 商品包 +（可选）优惠码，按业务线模版生成投放落地页链接，可预览/复制/删除。

二期交付物 / 验收

  • 商品包可上下架，下架后不可被券绑定（前置校验）。
  • 满减券可生成多优惠码、按 KOL 统计使用量；可调整可用商品、延长领取时间、停止发放（二次确认）。
  • 券与商品包通过 商品包ID 关联，币种与业务线一致。
  • 落地页可基于已配置业务线模版生成可用投放链接，并关联到对应渠道/商品包/优惠码。
后续能力增强（暂不在本次范围）规划中

会议纪要中提出但不在一 / 二期承诺范围的需求，记录在册、后续评估。

  • 用户标签体系 & 黑名单识别
  • 角色权限 / 数据权限（市场、销售、客服分权）
  • 邀约场景 & 提报成单率统计
  • 第三方整合：Breeze 消息推送、Apespire 流量进度
  • 商品品类丰富度扩展、活跃测试

1.1 渠道管理 一期 P0

维护标准化的渠道分级结构，并在任意层级生成实际投放使用的渠道 code，作为用户注册归因的唯一口径。

🆕
本期更新：渠道树新增最顶层「业务线」层级，结构调整为 业务线 → 渠道类型 → 一级 → 二级 → 三级；入口按钮改为「新增业务线」。渠道 code 生成规则改为 7 位随机字母数字（如 K2000Gh），不再带类型前缀。

用户故事

作为市场投放同学，我希望按「业务线 → 渠道类型 → 一级 → 二级 → 三级」维护渠道，并为具体投放位生成唯一 code，以便用户注册时能准确归因到业务线与来源。

用户故事

运营自助配置满减券：定义发放/领取规则、使用规则（绑定可用商品包）、满减权益，并支持编辑、延长、停止发放。

🏷️
二期功能：原型中本模块导航已标记「二期」，进入页面顶部带提示横幅「该模块为二期规划功能，当前页面仅作原型展示，相关功能将在二期上线」。
🆕
本期更新：一张券支持多个优惠码（按 KOL / 使用方）——为不同 KOL 各生成一个优惠码，分别统计已使用数量用于结算。生成 / 管理优惠码时可按 KOL 追加、批量复制（仅优惠码 / KOL+优惠码）；列表新增「优惠码」列与「管理优惠码」操作；详情含各码使用量及合计；搜索支持 券ID / 券名称 / 券码 / KOL。

用户故事

字段说明

关键业务规则

  • 一张券至少包含一个优惠码；同券内 KOL / 使用方不可重复。
  • 各优惠码的「已使用」数量独立统计，用于按 KOL 结算。
  • 抵扣金额不得大于门槛金额（表单校验）。
  • 可用商品必须为已存在的商品包，按商品包 ID 校验，不存在则报错。
  • 停止发放后不可继续领取；领取截止时间可向后延长以恢复「已生效」。
  • 券币种、业务线与所绑商品包须一致（前置约束）。
